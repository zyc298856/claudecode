# 工作报告写作风格指南

本文档规定工作报告的写作风格和用词规范，确保输出报告风格统一、专业规范。

## 核心原则

### 1. 简洁专业

- 每条事项一句话概括，避免冗长描述
- 不说废话，不写套话
- 删除"完成了"、"进行了"等冗余表达，直接写核心动作

**反面示例**：
- 进行了会议纪要功能的 bug 修复工作
- 完成了和导师关于 Q2 排期的沟通

**正面示例**：
- 修复会议纪要功能的 VTT 解析 bug
- 与导师确认 Q2 开发排期

### 2. 动词开头

每条事项以明确的动词开头，突出动作和成果。

**常用动词**：
- 开发、实现、完成、交付
- 修复、优化、重构、调整
- 设计、编写、制定、规划
- 测试、验证、部署、上线
- 调研、分析、评估、对比
- 沟通、协调、确认、对齐
- 参加、组织、主持

### 3. 数据驱动

优先使用量化表达，让成果可衡量。

**量化角度**：
- 数量：完成 N 个功能、修复 N 个 bug
- 比例：提升 N%、降低 N%、覆盖 N%
- 时间：耗时从 X 降至 Y、提前 N 天交付
- 规模：支持 N 并发、处理 N 条数据

**注意事项**：
- 只有用户提供的数据或上下文明确的数据才可使用
- 不要编造数据，如无数据则保留定性描述
- 可以使用"约"、"超过"、"近"等模糊修饰词

### 4. 客观描述

- 如实记录，不夸大不缩小
- 使用中性语言，不带主观评价
- 避免"非常"、"极大"、"显著"等程度副词（除非有数据支撑）

## 用词规范

### 事项描述

| 推荐用词 | 避免用词 |
|---------|---------|
| 开发 | 做了 |
| 修复 | 搞定了 |
| 优化 | 改了改 |
| 完成 | 弄好了 |
| 调研 | 看了看 |
| 协调 | 找人聊了聊 |

### 状态描述

| 状态 | 含义 |
|------|------|
| 已完成 | 工作已交付，无需后续跟进 |
| 进行中 | 当前正在推进 |
| 待处理 | 已知问题，尚未开始处理 |
| 已阻塞 | 因外部依赖无法继续推进 |

### 问题影响范围

| 范围 | 描述 |
|------|------|
| 局部 | 仅影响当前任务或模块 |
| 模块级 | 影响整个功能模块 |
| 项目级 | 影响项目整体进度 |
| 团队级 | 需要跨团队协调 |

## 常见问题

### 输入太散乱怎么办？

- 合并描述同一件事的多条记录
- 过滤寒暄、表情符号、无关信息
- 按时间或逻辑顺序整理
- 补充必要的上下文（如项目名称）

### 没有量化数据怎么办？

- 不要编造数据
- 保留定性描述，用清晰的动词表达
- 可以描述工作的范围和复杂度作为替代

### 如何处理模糊的时间表述？

- "这几天" → 列入本周总结
- "快完成了" → 标注"进行中"
- "应该没问题" → 标注"待验证"

### 用户输入混合了多个项目怎么办？

- 按项目/模块分组
- 每组独立一个小节
- 在小节标题中标注项目名称

### 英文输入怎么处理？

- 输出使用英文
- 动词使用过去式（已完成）或一般现在时（计划）
- 保持专业正式的语气
